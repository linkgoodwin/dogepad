致命根因已定位：createTokenForDao 函数不存在

LaunchDAO.sol 第619行调用 IBondingCurveLaunch(bondingCurve).createTokenForDao(...)，但 BondingCurve.sol 中根本没有实现这个函数。BondingCurve 只有 createToken 和 createTokenWithPresale 两个创建函数。

当用户认购累计达到 20 USDC (LAUNCH_THRESHOLD) 时，subscribeUsdc → _enqueueCandidate → _processQueueInternal 调用链被触发，在 _processQueueInternal 内部调用 createTokenForDao 时 整个交易 revert，导致认购交易本身也回滚。这就是"一到20u就失败"的直接原因。

🧠 Phase 1: 马斯克第一性原理分析

第一性原理拆解：什么是"代币发行"？

代币发行的本质 = 创建ERC20合约 + 初始分配 + 定价机制 + 流动性。当前系统把这四步分散在三个合约中，但缺少关键的粘合函数。

🎭 Phase 2: 女娲补充视角 — pump.fun & arc.network 创始人

多人物圆桌分析

Alon Cohen (pump.fun 创始人)

核心观点：pump.fun 的成功在于"一键发币 + bonding curve + graduation"的原子化流程。用户付一点钱就能发币，bonders 在曲线上买卖，当市值达到阈值时自动 graduate 到 DEX。整个流程零摩擦。

对 DogePad 的诊断：你们的流程在"graduation"环节断裂了。LaunchDAO 试图调用一个不存在的函数，相当于 pump.fun 的 graduation 合约里少写了 graduate 函数 — 任何达到阈值的币都会卡死。这不是优化问题，是基础功能缺失。

建议：1) 立即在 BondingCurve 中实现 createTokenForDao；2) 确保 listOnDex 真正添加流动性而非只设标记；3) pump.fun 的 graduation 是自动的，不需要等特定时间窗口。

Jeremy Allaire (Circle/arc.network 创始人)

核心观点：稳定币原生链的价值在于确定性结算。每笔交易都应该有确定性的结果 — 要么成功，要么失败，不应该出现"部分成功"或"静默失败"。

对 DogePad 的诊断：你们的 listOnDex 用 try/catch 静默吞掉错误（第663行），这违反了确定性原则。如果上DEX失败，用户应该能感知到，而不是以为币已经上了DEX但实际无法交易。另外，createTokenForDao 不存在导致 revert，这是"确定性失败"但发生在错误的地方 — 应该在编译期就发现这个问题。

建议：1) 所有关键操作不应该用 try/catch 静默处理；2) USDC 原生链上，msg.value 就是 USDC，不需要复杂的 wrapped token 逻辑；3) 确保接口定义和实现严格匹配。

共识区

  • 两人一致： createTokenForDao 缺失是致命问题，必须立即修复
  • 两人一致： listOnDex 只设标记不实际上DEX是严重设计缺陷
  • 两人一致： 发币流程应该是原子化的，不应该有时间窗口限制

A3. 时间限制分析

585 if ((block.timestamp % EPOCH_DURATION) / 1 hours < launchHour) return;

launchHour = 4，EPOCH_DURATION = 1 days = 86400。这意味着：

  • UTC 00:00 - 03:59：队列处理直接返回，不执行发币。认购交易可以成功（但不会发币）。
  • UTC 04:00 - 23:59：队列处理会执行，调用 createTokenForDao → REVERT。
注意：如果用户在 UTC 00:00-03:59 之间认购到20 USDC，交易会成功（因为时间限制让队列处理提前返回），但代币不会发行。到了 UTC 04:00 后，任何人调用 processQueue() 或再次认购时，才会触发 createTokenForDao 的 revert。

A4. 每日发币限制

584 if (dayLaunchCount[today] >= maxLaunchsPerDay) return; // maxLaunchsPerDay = 1

每天最多发1个币。如果今天已经发过币，队列处理会提前返回，不会触发 revert。但这意味着第二个达到阈值的候选者无法发行。

A5. 超额认购处理

B2. Bonding Curve 定价数学

文件：BondingCurveMath.sol

// 线性 bonding curve: price = basePrice + slope * tokensSold / PRECISION // basePrice = 100 gwei = 1e8 (0.0000000001 USDC per token) // slope = 10000 // PRECISION = 1e18 function calcBuyAmount(usdcAmount, tokensSold, basePrice, slope) { priceS0 = basePrice + (slope * tokensSold) / 1e18; // 解二次方程: (slope/2)*ds^2 + priceS0*ds - usdcAmount = 0 discriminant = priceS0^2 + 2 * slope * usdcAmount; sqrtD = sqrt(discriminant); ds = (sqrtD - priceS0) * 1e18 / slope; return ds; }

20 USDC 买入计算示例：

计算过程：
usdcIn = 20 ether = 20e18
fee = 20e18 * 300 / 10000 = 0.6 USDC
net = 19.4 USDC = 19.4e18
priceS0 = 1e8 (初始价格)
discriminant = 1e16 + 2 * 10000 * 19.4e18 = 1e16 + 3.88e23 ≈ 3.88e23
sqrtD ≈ 6.23e11
ds = (6.23e11 - 1e8) * 1e18 / 10000 ≈ 6.23e25
tokens ≈ 62,300 (以18位小数计)

BC 持有 75% 总供应量 = 750,000,000e18，远大于 62,300，所以 ICT() 不会触发。

B6. 内盘交易潜在问题

MEDIUM持仓限制可能阻止大额买入

当 dexPair == address(0)（未上DEX）时，每个地址最多持有 5% 流通量。初始流通量 = 总供应量 - BC余额 = 25% 总供应量（10% creator + 10% BC额外 + 5% feeDist）。5% * 25% = 1.25% 总供应量。

对于 10亿总供应量，单个地址最多持有 1250万代币。如果用户用大量 USDC 买入，可能触发 "exceeds holding limit" 错误。

修复建议：对于 LaunchDAO 的买入，确保先调用 excludeFromHoldingLimit(LaunchDAO)。对于普通用户，5% 限制是合理的防鲸鱼机制。
LOW推荐奖励可能导致 ICT 错误

第233-238行：推荐奖励额外铸造代币给推荐人。如果 BC 余额不足以支付推荐奖励，会静默跳过（不 revert）。这不是严重问题，但推荐人可能拿不到奖励。

严重问题：listOnDex 只是把 i.listed 设为 true，没有调用 DexLister.addLiquidityAndDistribute() 来添加流动性。这意味着：

C5. SimpleRouter DEX 交易逻辑

文件：SimpleRouter.sol

SimpleRouter 是一个标准的 Uniswap V2 风格的 AMM 路由器，支持：

  • addLiquidity / addLiquidityETH — 添加流动性
  • swapExactTokensForTokens — 代币换代币
  • swapExactETHForTokens — USDC 买代币
  • swapExactTokensForETH — 代币卖 USDC

手续费：0.3%（amountIn * 997 / 1000）

SimpleRouter 本身逻辑正确，但前提是流动性池已经创建。由于 listOnDex 没有调用 DexLister，流动性池永远不会被创建。

C6. BCT 税收在外盘的行为

当 dexPair 被设置后，BCT 的 _update 会识别 DEX 交易：

// 买入（从 DEX pair 转出） bool isBuy = (from == bondingCurve || from == address(this)) || (dexPair != address(0) && from == dexPair); // ← DEX 买入 // 卖出（转入 DEX pair） bool isSell = (to == bondingCurve || to == address(this)) || (dexPair != address(0) && to == dexPair); // ← DEX 卖出

DEX 买入收 3% 税，DEX 卖出收 5% 税。税收发送到 taxReceiver（feeDistributor）。

但 dexPair 只有在 DexLister.addLiquidityAndDistribute() 中通过 setDexPair(lpToken) 设置。由于这个函数从未被调用，dexPair 永远是 address(0)。
